# 验收标准：MVP — 核心骨架与单节点闭环

> 本文档只定义**做到什么算完成**，不规定怎么做——实现方法是执行层的事。
> 范围 = 最核心的骨架搭建 + 单节点 agent 闭环。架构见《白皮书》，契约见《插件接口设计》。
>
> **迁移声明**：本 MVP 是对现有代码（`server/` 的 Python 实现、`core/` 的 Rust 实现、`client/` 的 Vue 前端）向新架构的迁移重构，**不是从零新建**。涉及的现有逻辑复用与废弃方向见《技术架构选型》第 10 章与《官方插件设计》附录。本文档不含迁移步骤。

---

## 0. MVP 的边界

**做到**：在**单台机器**上打开 UI，与 AI 对话，AI 能读文件、写文件、跑命令，多轮工具调用闭环，结果在 UI 渲染。

**MVP 明确不做**（后续阶段）：libp2p 网络/节点发现、Gossip 同步、资源寻址的跨节点路由（单节点下路由平凡）、能力令牌、融合 LLM（多 provider/调度）、记忆、Spec、并发控制、跨域、移动端。

单节点是 MVP 的运行约束；但骨架（六机制的最小可用形态 + 统一注册 + 管道引擎）必须按"将来能长成分布式"的形态搭建，不能写成单机死结构。

---

## 1. 内核骨架

### AC-1.1 点注册表
- [ ] 存在固定词汇表的点：`user-message`、`llm-prompt`、`llm-response`、`tool-execute`(含 `.before`/`.after`)、`tool-result`、`tool:*`、`ui:*`、`lifecycle:*`。
- [ ] handler 可按《插件接口设计》第 3 节的 manifest 注册到点上（声明：point、entry、filter、before/after、transport）。
- [ ] 注册时拒绝"发明新点"的请求（挂到词汇表外的点 → 报错）。
- [ ] **验收方式**：注册两个 handler 到同一点并声明 before/after，查询该点返回拓扑序正确的 handler 列表；尝试注册一个未知点 → 收到明确拒绝错误。

### AC-1.2 管道引擎
- [ ] 流经过一个点时，按 before/after 拓扑序依次调用该点的 handler，传入 ctx、接收返回的 ctx。
- [ ] 某 handler 置 `ctx.terminated = true` → 后续 handler 不再执行。
- [ ] **agent 循环控制流**：`llm-response` 含 toolcall → 执行 → 结果喂回 → 回到 `llm-prompt` 继续；不含 toolcall → 终止循环。
- [ ] **验收方式**：构造一个会连续调用 2 次 toolcall 后给出文本回复的场景，闭环正确终止；ctx 在各点间字段传递无丢失。

### AC-1.3 统一注册与 transport 适配
- [ ] handler 通过统一 `Handler` 接口被调用，调用方不感知 transport。
- [ ] MVP 至少支持 **native** transport（官方插件进程内）。gRPC/MCP 接口预留但不要求在 MVP 实现。
- [ ] **验收方式**：所有 MVP 官方插件以 native 注册并被管道引擎统一调用，调用处代码不出现 transport 分支。

## 2. 最小核插件

### AC-2.1 朴素 LLM 调用
- [ ] 挂 `llm-prompt`→`llm-response`，对接一个 OpenAI/Anthropic 兼容端点。
- [ ] 能发送 prompt（含 system + 历史 + 工作区目录树）、接收响应、解析出文本或 toolcall。
- [ ] 不含多 provider/fallback/调度（那是融合 LLM 升级版）。
- [ ] **验收方式**：配置一个模型端点，发一句话能拿到回复；回复中的 toolcall 被正确解析成结构化调用。

### AC-2.2 基础工具
- [ ] 注册 `tool:file_read`、`tool:file_write`、`tool:shell`，schema 符合 OpenAI/Anthropic function calling 规范。
- [ ] 路径参数为工作区相对路径，缺省解析到当前工作目录。
- [ ] file_write 创建不存在的父目录；shell 捕获 stdout/stderr 与超时。
- [ ] **验收方式**：AI 通过对话成功读取一个文件、写入一个新文件（含新目录）、执行一条 shell 命令并拿到输出。

### AC-2.3 聊天面板
- [ ] UI 主区渲染消息流（用户/助手）、流式输出、toolcall 折叠/展开、diff、输入框。
- [ ] 遵循《前端设计规范》token（四套主题、去卡片化、lucide 图标）。
- [ ] **验收方式**：在 UI 输入指令，看到流式回复、工具调用过程、文件 diff，视觉符合设计规范。

## 3. 工作区（单节点）

### AC-3.1 工作区映射与注入
- [ ] 支持打开本地文件夹作为工作区挂载（单节点下 Owner = 本机）。
- [ ] 工作区目录树作为上下文注入 prompt，使模型能传出正确的相对路径。
- [ ] toolcall 相对路径经工作区映射解析到具体文件（单节点下平凡解析到本机）。
- [ ] **验收方式**：打开一个项目文件夹，AI 能正确理解目录结构并按相对路径读写其中的文件。

### AC-3.2 骨架为分布式预留
- [ ] 工作区映射的数据结构为"挂载 → 项目 → Owner 节点"形态（单节点下 Owner 恒为本机），不写成"路径 = 本地绝对路径"的死结构。
- [ ] **验收方式**：代码审查确认工作区/路由抽象具备承载多 Owner 的形态，无需重写即可在网络层阶段接入远程 Owner。

## 4. 整体闭环验收

### AC-4.1 端到端
- [ ] 在单机打开 UI → 打开一个项目 → 用自然语言要求 AI 完成一个涉及"读现有文件 + 写修改 + 跑一次命令验证"的小任务 → AI 多轮 toolcall 闭环完成 → 结果与 diff 在 UI 正确呈现。
- [ ] **这等价于"你和 AI 对话改代码"的完整体验**，是 MVP 的总验收。

### AC-4.2 架构纪律（防止写歪）
- [ ] 内核不含任何具体 AI 逻辑/工具实现/LLM 实现——它们全部是注册到点上的 handler。
- [ ] 不存在内核无法验证的自报标签（无 `affinity`/`stateful`/`exec` 之类）。
- [ ] 官方插件与（未来）社区插件走同一注册流程，无特权后门。
- [ ] **验收方式**：代码审查确认内核目录无业务逻辑；插件清单声明字段仅含可验证项 + 幂等性。
