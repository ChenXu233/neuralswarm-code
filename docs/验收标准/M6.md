# M6: 调度 + 安全

**目标**：调度器自动拆解任务并匹配最优 Agent+LLM 组合，安全层保障通信和权限。

**可见产出**：提交复杂任务后自动拆解为子任务，分配给最合适的 Agent+LLM。

**并行轨道**：
- Track A（服务器）：调度器核心 + 安全层
- Track C（UI）：调度决策可视化

---

## 阶段 6.1 调度器核心

**目标**：基于 DAG 的任务拆解和 Agent+LLM 能力匹配。

**验收任务**：
- [ ] **任务解构**：使用轻量 LLM 将用户任务分解为子任务 DAG
  - 输入：用户任务描述
  - 输出：子任务列表 + 依赖关系（DAG）
  - 每个子任务标注所需能力（工具集、推理等级）
- [ ] **Agent+LLM 能力匹配**：
  - Agent 能力：工具集覆盖度
  - LLM 能力：推理/代码生成等级
  - 匹配评分函数：`Score(A, L, T) = agent_match(A, T) * llm_match(L, T) * g(load_A)`
- [ ] **LLM 自动切换**：
  - 限流检测（429 → 标记 throttled）
  - 按能力等级降级
  - Token 消耗均衡（优先选择消耗较低的模型）
  - 恢复探测（定期探测被限流的 LLM）
- [ ] **DAG 执行引擎**：
  - 按依赖顺序执行子任务
  - 无依赖的子任务并行执行
  - 失败处理（重试 / 降级 / 通知用户）
- [ ] 调度状态 API：`GET /api/scheduler/tasks/{id}/dag`
- [ ] 单元测试覆盖

**验收实践**：
```python
# 提交复杂任务
task = await scheduler.submit(
    "给项目添加用户认证功能，包括注册、登录、JWT 中间件"
)

# 预期：自动拆解为 DAG
# 子任务 1：设计认证架构（无依赖）
# 子任务 2：实现注册接口（依赖 1）
# 子任务 3：实现登录接口（依赖 1）
# 子任务 4：实现 JWT 中间件（依赖 1）
# 子任务 5：集成测试（依赖 2, 3, 4）

# 调度器自动匹配：
# 子任务 1 → Agent-Review + Claude Opus（需要强推理）
# 子任务 2-4 → Agent-Code + Claude Sonnet（代码生成）
# 子任务 5 → Agent-Test + DeepSeek（测试，成本低）
```

## 阶段 6.2 安全层

**目标**：mTLS 加密通信，RBAC 权限控制，操作审计。

**验收任务**：
- [ ] **mTLS 证书**：
  - 自动生成自签名 CA
  - 服务器和客户端证书生成
  - 证书验证（连接时校验证书链）
- [ ] **RBAC 中间件**：
  - 角色定义：admin、operator、viewer
  - 权限矩阵：
    | 操作 | admin | operator | viewer |
    |------|-------|----------|--------|
    | 提交任务 | ✅ | ✅ | ❌ |
    | 查看任务 | ✅ | ✅ | ✅ |
    | 管理 Agent | ✅ | ❌ | ❌ |
    | 锁决策 | ✅ | ✅ | ❌ |
  - API 端点权限标注
- [ ] **操作审计日志**：
  - 结构化日志（JSON 格式）
  - 记录：agent_id、action、resource、detail、timestamp
  - 不可篡改（append-only）
  - 审计日志 API：`GET /api/audit?agent_id=xxx`
- [ ] 单元测试覆盖
