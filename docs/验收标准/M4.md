# M4: 并发

**目标**：多个 Agent 并发执行任务，worktree 隔离 + 哈希并发控制，冲突时用户可决策。

**可见产出**：多个 Agent 同时工作，文件冲突时 UI 弹窗让用户选择（重新阅读 / 覆盖 / 仲裁）。

**并行轨道**：
- Track A（服务器）：Central Scheduler + Agent 池 + Worktree 管理 + 哈希并发控制
- Track C（UI）：Agent 状态面板 + 冲突弹窗 + Sub-scheduler 对话

---

## 阶段 4.1 Central Scheduler 框架

**目标**：Agent 池管理，支持多 Agent 并发执行不同任务。

**验收任务**：
- [ ] AgentPool 管理器（`AgentPool`）：
  - 创建 Agent（绑定 task_id、worktree_path、工具集、LLM 配置）
  - 销毁 Agent（任务完成或失败）
  - 查询 Agent 状态
- [ ] CentralScheduler 核心逻辑：
  - `submit_task(task)` → 创建 worktree + Sub-scheduler
  - `allocate_resources(request)` → 分配 Worker Agents
  - `complete_task(task_id)` → 合并 worktree + 清理
- [ ] Agent 状态机（IDLE → PLANNING → RUNNING → COMPLETED / FAILED）
- [ ] 并发数控制（`max_concurrent_tasks`，可配置）
- [ ] Agent 状态查询 API：`GET /api/agents`
- [ ] 单元测试覆盖

**验收实践**：
```bash
# 同时提交 5 个任务
for i in {1..5}; do
  curl -X POST http://localhost:8000/api/tasks \
    -H "Content-Type: application/json" \
    -d "{\"prompt\": \"任务 $i\"}" &
done
wait

# 查询 Agent 状态
curl http://localhost:8000/api/agents
# 预期：5 个 Agent 并发执行，各自独立 worktree
```

## 阶段 4.2 Worktree 管理

**目标**：每个 Task 一个 Git worktree，隔离执行环境。

**验收任务**：
- [ ] WorktreeManager（`WorktreeManager`）：
  - `create(project_id, task_id)` → 创建 worktree
  - `remove(task_id)` → 删除 worktree
  - `merge(task_id)` → 合并到主分支
  - `list()` → 列出所有 worktree
- [ ] Git worktree 集成（`git worktree add/remove`）
- [ ] 冲突检测（Git merge conflict）
- [ ] Worktree 清理（Task 完成/失败后自动清理）
- [ ] 单元测试覆盖

**验收实践**：
```python
# 创建 worktree
worktree = await worktree_manager.create("project-01", "task-01")
assert os.path.exists(worktree.path)

# 合并 worktree
await worktree_manager.merge("task-01")

# 清理
await worktree_manager.remove("task-01")
assert not os.path.exists(worktree.path)
```

## 阶段 4.3 Sub-scheduler LLM

**目标**：Sub-scheduler 对话式规划，自己执行简单任务，申请 Worker 执行复杂任务。

**验收任务**：
- [ ] SubSchedulerAgent（`SubSchedulerAgent`）：
  - `start(task)` → 分析任务 + 对话规划
  - `execute_plan()` → 执行计划（自己做 or 申请 Worker）
  - `request_workers(subtasks)` → 向 Central Scheduler 申请 Worker
  - `coordinate(workers)` → 协调 Worker 执行
- [ ] WebSocket 对话端点：`/ws/chat`
- [ ] 任务拆分逻辑（LLM 分析 → 子任务列表）
- [ ] 资源申请协议：`POST /api/scheduler/allocate`
- [ ] 单元测试覆盖

**验收实践**：
```bash
# 提交任务 → Sub-scheduler 对话
curl -X POST http://localhost:8000/api/tasks \
  -H "Content-Type: application/json" \
  -d '{"prompt": "重构 src/main.py，提取工具函数"}'

# WebSocket 连接查看对话
# 预期：Sub-scheduler 分析任务，与用户确认计划
```

## 阶段 4.4 哈希并发控制

**目标**：多 Agent 共享 worktree 时，通过文件哈希防止同时修改冲突。

**验收任务**：
- [ ] HashGuard（`HashGuard`）：
  - `read(file_path)` → 读取文件 + 记录哈希
  - `write(file_path, content, agent_id)` → 写入前校验哈希
  - 哈希变化 → 返回冲突（不写入）
- [ ] WorkerAgent 集成 HashGuard：
  - 读文件 → 记录哈希
  - 修改文件 → 写入前校验
  - 冲突 → 通知 ConflictManager
- [ ] Git commit 集成（每次写入后自动提交）
- [ ] 单元测试覆盖

**验收实践**：
```python
# 模拟两个 Agent 同时修改同一文件
guard = HashGuard(worktree_path)

# Agent-1 读取
content1 = await guard.read("src/main.py")
# Agent-2 读取
content2 = await guard.read("src/main.py")

# Agent-1 写入（成功）
result1 = await guard.write("src/main.py", "new content 1", "agent-1")
assert result1 == True

# Agent-2 写入（冲突）
result2 = await guard.write("src/main.py", "new content 2", "agent-2")
assert result2 == False  # 哈希变化，冲突
```

## 阶段 4.5 冲突管理

**目标**：冲突时通过 WebSocket 通知用户，用户可决策。

**验收任务**：
- [ ] ConflictManager（`ConflictManager`）：
  - `detect(conflict)` → 检测冲突 + WebSocket 通知
  - `resolve(conflict_id, action)` → 执行用户决策
- [ ] Conflict 数据模型（`Conflict`）
- [ ] WebSocket 端点：`/ws/conflicts`
- [ ] 冲突决策 API：`POST /api/conflicts/{id}/decide`
  - `re_read`：Agent 重新阅读文件，基于最新内容重做
  - `overwrite`：强制用 Agent 的内容覆盖
  - `submit_to_scheduler`：展示 diff，Scheduler 仲裁
- [ ] 超时自动执行默认策略（可配置，默认 re_read）
- [ ] 单元测试覆盖

**验收实践**：
```python
# 检测到冲突
conflict = Conflict(
    file_path="src/main.py",
    agent_id="agent-2",
    other_agent_id="agent-1",
    old_hash="abc",
    current_hash="def",
    current_content="...",
    new_content="...",
)
await conflict_manager.detect(conflict)
# 预期：WebSocket 推送冲突通知

# 用户决策
await conflict_manager.resolve(conflict.id, "re_read")
# 预期：Agent 重新读取文件，基于最新内容重做
```

## 阶段 4.6 UI 并发视图

**目标**：UI 展示多 Agent 状态和冲突决策界面。

**验收任务**：
- [ ] Agent 状态面板（列表/卡片）：
  - 显示每个 Agent 的类型（Sub-scheduler / Worker）
  - 显示 Agent 状态（IDLE / PLANNING / RUNNING / COMPLETED / FAILED）
  - 显示 Agent 关联的 Task
  - 点击 Agent 查看对话/执行详情
- [ ] 冲突弹窗：
  - 显示冲突文件路径
  - 显示冲突双方（Agent-1 已修改 vs Agent-2 想修改）
  - 展示 diff（当前内容 vs 新内容）
  - 决策按钮（重新阅读 / 覆盖 / 仲裁）
  - 倒计时（超时自动执行默认策略）
- [ ] Sub-scheduler 对话界面：
  - 显示 Sub-scheduler 的规划过程
  - 显示任务拆分结果
  - 用户确认/修改计划
- [ ] 任务队列显示（等待中的任务）
