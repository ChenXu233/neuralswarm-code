# M4: 并发

**目标**：多个 Agent 并发执行任务，资源锁保证并发安全，锁冲突时用户可决策。

**可见产出**：多个 Agent 同时工作，锁冲突时 UI 弹窗让用户选择（抢占/排队/合并/手动）。

**并行轨道**：
- Track A（服务器）：多 Agent 池 + 分布式锁
- Track C（UI）：锁冲突通知 + 多 Agent 视图

---

## 阶段 4.1 多 Agent 并发

**目标**：Agent 池管理，支持多 Agent 并发执行不同任务。

**验收任务**：
- [ ] Agent 池管理器（`AgentPool`）：
  - 创建 Agent（绑定 project_id、工具集、LLM 配置）
  - 销毁 Agent（任务完成或超时）
  - 回收 Agent（重置状态，等待新任务）
- [ ] 多 Agent 并发执行（asyncio Task 池）
- [ ] Agent 生命周期管理（idle → running → completed → recycled / failed → destroyed）
- [ ] 并发数控制（`max_concurrent_tasks`，可配置）
- [ ] Agent 状态查询 API：`GET /api/agents`
- [ ] 单元测试覆盖

**验收实践**：
```bash
# 同时提交 5 个任务
for i in {1..5}; do
  curl -X POST http://localhost:8000/api/tasks \
    -H "Content-Type: application/json" \
    -d "{\"prompt\": \"任务 $i\"}" &
done
wait

# 查询 Agent 状态
curl http://localhost:8000/api/agents
# 预期：5 个 Agent 并发执行，无死锁
```

## 阶段 4.2 分布式锁

**目标**：基于 Redis 的 Redlock 实现，保护共享资源。

**验收任务**：
- [ ] Redlock 算法实现（redis-py）
- [ ] 锁模式：
  - **X 锁**（独占）：获取即排他
  - **S 锁**（共享读）：多读并发
  - **IX/IS 锁**（意向）：目录级锁
- [ ] URI 资源标识（`project://xxx/src/utils.py`）
- [ ] 锁超时和自动释放（防止死锁）
- [ ] 锁信息持久化到 PostgreSQL（崩溃恢复）
- [ ] 锁状态查询 API：`GET /api/locks`
- [ ] 单元测试覆盖

**验收实践**：
```python
# 1. 获取独占锁
from neuralswarm.services.lock_manager import LockManager

lock_mgr = LockManager(redis_client, db_session)
lock = await lock_mgr.acquire(
    resource="project://test/src/main.py",
    mode="X",
    agent_id="agent-01"
)
# 预期：获取成功

# 2. 冲突测试
lock2 = await lock_mgr.acquire(
    resource="project://test/src/main.py",
    mode="X",
    agent_id="agent-02"
)
# 预期：获取失败，返回冲突信息（holder_agent_id, acquired_at）

# 3. 释放锁
await lock_mgr.release(lock)
# 预期：释放成功

# 4. 锁超时测试
# 获取锁后不释放，等待超时
# 预期：锁自动释放
```

## 阶段 4.3 锁冲突通知

**目标**：锁冲突时通过 WebSocket 通知用户，用户可决策。

**验收任务**：
- [ ] WebSocket 端点 `/ws/conflicts`
- [ ] 锁冲突时推送 `scheduling_request` 事件：
  ```json
  {
    "type": "scheduling_request",
    "data": {
      "conflict_id": "conflict-001",
      "resource": "project://test/src/main.py",
      "holder": {"agent_id": "agent-01", "task": "重构 main.py", "held_since": "..."},
      "requester": {"agent_id": "agent-02", "task": "修复 bug #42"},
      "actions": ["preempt", "queue", "merge", "manual"],
      "recommended": "queue",
      "timeout_seconds": 300
    }
  }
  ```
- [ ] 用户决策接口：`POST /api/conflicts/{id}/decide`
  - `preempt`：保存 holder 检查点，终止 holder，锁交给 requester
  - `queue`：requester 加入等待队列
  - `merge`：将 requester 任务注入 holder 一并执行
  - `manual`：暂停自动操作，等待用户手动处理
- [ ] 超时自动执行默认策略（可配置，默认 queue）
- [ ] 单元测试覆盖

## 阶段 4.4 UI 并发视图

**目标**：UI 展示多 Agent 状态和锁冲突决策界面。

**验收任务**：
- [ ] 多 Agent 状态面板（列表/卡片，显示每个 Agent 的状态、任务、LLM）
- [ ] 锁冲突弹窗：
  - 显示冲突详情（资源、holder、requester）
  - 显示推荐操作
  - 决策按钮（抢占/排队/合并/手动）
  - 倒计时（超时自动执行）
- [ ] 任务队列显示（等待中的任务）
- [ ] Agent 对话流切换（点击不同 Agent 查看各自的对话）

```bash
#   - UI 弹出锁冲突通知

#   - Agent A 完成后，Agent B 自动获得锁继续执行
```
