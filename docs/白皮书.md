# NeuralSwarm/Code 白皮书 v0.7

**一个面向项目型代码协作的异构多智能体 Mesh 调度系统**

## 1. 摘要

**NeuralSwarm/Code** 是一个异构多智能体（Heterogeneous Multi-Agent）Mesh 调度系统，它将”神经”（Neural）的快速原生连接与”群蜂”（Swarm）的协同智能注入”代码”（Code）这一核心项目场景。它部署于服务器，通过客户端-服务器架构形成 **Mesh 网络拓扑**：客户端之间可通过服务器路由互访资源，服务器之间亦可对接其他 Mesh 网络，实现跨实例互联。

**核心概念区分**：

- **Agent（智能体）**：有状态的运行实例，持有上下文、记忆、工具集，绑定项目和任务。Agent 是调度和执行的单位。
- **LLM（大语言模型）**：无状态的推理后端，被 Agent 调用。多个 Agent 可共享同一个 LLM，一个 Agent 在执行过程中可切换 LLM（如限流时降级）。
- **调度 = 选 Agent + 选 LLM**：调度器同时考虑 Agent 的工具能力和 LLM 的推理能力，做组合匹配。

**核心差异化能力矩阵**：

- **Mesh 网络拓扑**：客户端安装 `neuralswarm-agent`，通过 SSH 反向隧道接入服务器。任意客户端可经由服务器 MCP 桥接网关访问其他客户端的资源，服务器可对接外部 Mesh 网络，形成蜂群式互联。
- **原生基础设施互联**：基于 MCP（Model Context Protocol）统一资源访问与原生 SSH 多级调用，客户端可如 Codex 般直接、可观测地控制宿主机。
- **项目级并发控制**：为代码资源（文件、数据库表）实现独占/共享锁，并提供**用户窗口级调度器**，将人工决策置于自动化工作流的回路中（Human-in-the-loop）。
- **异构能力分级与调度**：对 LLM 进行声明式能力配置，结合在线榜单校准，由中央调度器实现基于任务复杂度与 Agent/LLM 能力的动态最优匹配。支持模型限流自动切换与跨模型 token 消耗均衡。
- **项目隔离的认知架构**：为每个项目提供独立的统一 Spec 环境与多层记忆（瞬时/短期/长期/全局），确保上下文纯净与长期演进。

**外部依赖**（对接，不自建）：

- **模型网关**：对接任意兼容 OpenAI/Claude 格式的模型网关（如 ccswitch、OneAPI、NewAPI 等），负责多 LLM 提供商的统一路由。NeuralSwarm 适配 OpenAI 格式（`/v1/chat/completions`）和 Claude 格式（`/v1/messages`）调用模型网关，网关负责底层模型切换。
- **MCP 协议**：遵循 MCP 标准对接工具和资源服务，NeuralSwarm 不实现 MCP 协议本身，只做对接。

**参考项目**：

- **openclaw**：全局聊天模式的交互理念参考——个人助手、无固定项目绑定、可跨项目派发、不阻塞聊天任务。NeuralSwarm 自建全局聊天模块，与调度器、记忆系统、项目空间深度集成。

## 2. 动机与背景

### 2.1 现有AI Agent系统的结构性局限

2025年至今，AI Agent框架虽蓬勃发展，但在解决真实世界复杂工程问题时，暴露出结构性不足：

- **协作模式单一**：多数框架本质是单Agent的任务执行器，或仅支持集中编排的多Agent流水线（如LangGraph, CrewAI），缺乏去中心化的、对等的Mesh协作能力，存在单点故障和扩展瓶颈。
- **资源访问浅层化**：SSH、数据库、本地文件等被视为“外部工具”，通过函数调用进行交互。这丧失了原生会话管理、连接复用和跨端资源（如“服务器Agent读取客户端未提交的Git变更”）的统一抽象能力。
- **并发控制与人工脱节**：多Agent并发操作同一资源时，缺乏标准化的锁机制和冲突解决策略。当冲突发生时，系统只能自动失败或执行简单重试，无法将决策权实时、上下文丰富地移交给开发者。
- **上下文与记忆污染**：项目和Agent之间缺乏严格的逻辑隔离。全局记忆和共享上下文易导致信息泄露、指令冲突和“幻觉”级联放大。
- **能力的非定量匹配**：Agent 的工具能力和 LLM 的推理能力描述停留在”强/弱”或角色标签，无法根据任务的数学推理、长上下文、特定领域知识等量化需求，进行 Agent 和 LLM 的精细化组合匹配。

### 2.2 NeuralSwarm/Code 的设计哲学

**群体智能（Swarm Intelligence） × 原生基础设施（Infrastructure-as-Code） × 项目中心（Project-Centric）**

- **Mesh 拓扑**：客户端通过 SSH 反向隧道接入服务器，形成 Mesh 网络。任意客户端可经由服务器 MCP 桥接网关访问其他客户端的资源，服务器可对接其他 Mesh 网络。这是真正的蜂群互联，而非星型拓扑。
- **去中心化与集中式协同**：Agent 是独立的计算单元，能点对点委托和通过事件驱动协作，但关键任务分配和冲突解决由高可用的中央调度器仲裁。这是灵活性与可控性的平衡。
- **基础设施即一等公民**：SSH、文件系统、数据库等不再外挂，而是通过 MCP 桥接器成为系统**拓扑图**中的节点。代码项目是该拓扑的核心，所有资源和 Agent 围绕它组织。
- **对接不自建**：模型路由交给外部模型网关（ccswitch、OneAPI、NewAPI 等），MCP 协议遵循标准不自造。NeuralSwarm 专注于调度、锁、记忆和 Mesh 路由——这些才是核心价值。
- **认知闭环**：系统通过**多层记忆**和**统一 Spec 环境**为 Agent 构建了一个持续学习的”海马体”与”前额叶”，使其决策随项目演进不断优化。

## 3. 系统拓扑与架构

NeuralSwarm/Code 采用客户端-服务器 Mesh 架构。服务器是路由中枢，客户端通过 SSH 反向隧道接入，客户端之间可经由服务器互访资源，服务器可对接其他 Mesh 网络。


### 模块职责详述

- **中央调度器集群**：系统的”前额叶”。负责无状态的任务接收、复杂度分析、Agent 和 LLM 的组合能力匹配、Agent 生命周期管理。采用 etcd 进行领导者选举，实现高可用。
- **Agent 执行池**：系统的”肢体”。管理异构 Agent 实例——每个 Agent 持有自己的上下文、记忆、工具集和项目绑定，通过模型网关调用 LLM 作为推理后端。Agent 是有状态的、可被抢占和回收的进程。
- **模型网关对接层**：适配 OpenAI 格式和 Claude 格式，对接任意兼容的模型网关（ccswitch、OneAPI、NewAPI 等）。负责限流检测、自动切换和 token 消耗均衡。Agent 调用 LLM 时，请求经由此层路由到模型网关。
- **MCP 桥接网关**：系统的”脊髓”。它是一个智能的多路复用 MCP 客户端，负责将 Agent 的标准资源请求路由到正确的资源节点（包括跨客户端路由），并处理连接管理、重试和背压。同时负责与其他 Mesh 网络的互联路由。
- **分布式锁管理器**：系统的”基底神经节”。为项目范围内的 URI 资源提供高并发、高一致性的锁服务，并实现一个等待队列与用户交互的**调度窗口**。
- **认知与状态层**：系统的”记忆与知识”。项目 Spec 和向量化的多层记忆为 Agent 提供情境化的长期背景。

## 4. Agent 与 LLM：概念分离

### 4.1 Agent 与 LLM 的关系

Agent 和 LLM 是两个独立的概念，必须严格区分：

| 维度 | Agent（智能体） | LLM（大语言模型） |
|------|----------------|------------------|
| 状态 | 有状态（上下文、记忆、任务进度） | 无状态（输入→输出） |
| 生命周期 | 创建→执行→完成/终止 | 持久存在 |
| 能力来源 | 工具集、项目上下文、记忆 | 推理、代码生成、知识 |
| 可替换性 | 绑定项目和任务 | Agent 可在执行中切换 LLM |
| 共享关系 | 每个 Agent 独立 | 多个 Agent 可共享同一个 LLM |

**调度 = 选 Agent + 选 LLM**：调度器同时考虑 Agent 的工具能力（能否访问文件、执行 shell、操作 git）和 LLM 的推理能力（代码生成等级、长上下文支持），做组合匹配。

### 4.2 Agent 发现与能力注册

Agent 实例启动时，必须向调度器集群注册一个包含以下信息的**能力清单**。注意：Agent 能力和 LLM 能力是分开声明的——Agent 有工具和项目上下文，LLM 有推理和生成能力。

```json
{
  "agent_id": "agent-python-srv-01",
  "project_id": "proj-neuralswarm",
  "agent_capabilities": {
    "tools": ["file_read", "file_write", "shell", "git"],
    "project_context": true,
    "memory_access": ["L0", "L1", "L2"],
    "max_concurrent_tasks": 2
  },
  "llm": {
    "model_id": "claude-sonnet-4-6",
    "gateway_route": "claude-sonnet",
    "capabilities": {
      "codegen": 4,
      "reasoning": 4,
      "long_context": 4,
      "max_tokens": 200000
    },
    "rate_limit": {"rpm": 60, "tpm": 100000},
    "cost_tier": "medium",
    "is_free_tier": false
  },
  "tags": ["python", "pytorch", "postgres"],
  "mcp_session_id": "sess-xyz",
  "status": "idle"
}
```

能力分级采用**声明式配置 + 在线榜单校准**：
- 用户在配置文件中声明每个 LLM 的能力标签和等级（与 Agent 的工具能力分开）
- 调度器定期拉取在线基准榜单（如 LMSYS Chatbot Arena、HumanEval、MMLU 等）校准 LLM 能力分数
- 不做自动影子模式评估——用户比任何自动系统更了解自己的模型

### 4.3 Agent 工具与沙箱

Agent 的工具集不仅限于预定义工具，还支持**动态注册**和**自写工具**：

- **动态注册**：工具通过 `ToolExecutor.register(name, func)` 动态注册，Agent 可在运行时扩展能力。
- **自写工具**：Agent 可以通过 LLM 生成新的工具代码，并注册到 ToolExecutor 中。这使 Agent 能够自主扩展能力，适应未知任务。
- **沙箱执行**：自写工具必须在沙箱中执行，限制文件系统访问、网络访问和系统调用。沙箱提供安全隔离，防止恶意或错误代码影响系统。

**工具执行流程**：
```
LLM 输出工具调用 → ToolExecutor 查找 → 已注册工具直接执行
                                        → 未注册工具：LLM 生成实现 → 沙箱执行 → 注册复用
```

### 4.4 三种协作模式

1.  **集中编排（Orchestrated）**：默认模式。调度器将复杂任务拆解为有向无环图（DAG），匹配最优Agent并行/串行执行。这保证了过程的可预测性与可审计性。
2.  **点对点委托（Delegated）**：Agent执行中，若发现需要专有能力，可通过内部消息总线（经由调度器权限检查）向另一个Agent发起委托请求。此模式允许涌现出专家Agent，降低调度器中心负载。
3.  **事件驱动协作（Event-Driven）**：Agent可向事件总线发布标准事件（如 `file:changed:src/utils.py`），其他Agent可声明式订阅。适用于代码检查、实时文档更新等解耦场景。

### 4.4 SSH多级调用的拓扑实现

SSH在NeuralSwarm/Code中是**构建安全网络拓扑**的工具，而非简单的命令通道。

- 客户端安装的`neuralswarm-agent`会主动向服务器MCP桥接网关建立**SSH反向隧道**，将本地MCP服务端口映射到网关的特定路由前缀（如`client://zhang-laptop/*`）。
- 当服务端Agent A需要读取客户端文件时，它发起调用`bridge.read(“client://zhang-laptop/home/project/main.py”)`。
- MCP桥接网关将此请求通过隧道转发，客户端本地代理执行后返回。整个过程对Agent A完全透明，如同访问本地资源。用户的**悬浮窗**将可视化此完整调用链，并可随时中断。

## 5. MCP桥接网关：统一资源访问

MCP桥接网关是系统的资源抽象层，设计为高性能的异步IO网关。

- **声明式路由配置**：通过YAML定义资源前缀到MCP Server端点的映射，支持负载均衡和故障转移。
- **协议适配与增强**：除了标准MCP调用，桥接网关还负责注入安全上下文（`X-Agent-ID`， `X-Project-ID`）和追踪信息（`traceparent`），使后端服务能执行精细的访问控制与审计。
- **连接生命周期管理**：对每个后端连接池进行健康检查，支持指数退避重连。对于SSH隧道，它管理着隧道的创建、保活与优雅关闭。

## 6. 项目生态与并发控制

### 6.1 项目：一等公民的边界上下文

在DDD（领域驱动设计）中，项目是一个有明确边界的上下文。项目分为两种类型：

**云端项目**：存在于服务器磁盘上的目录，任何客户端（浏览器、桌面端、移动端）均可访问。Agent 直接读写服务器上的文件。

**本地项目**：存在于用户电脑上的目录，通过 SSH 隧道让云端 Agent 访问。文件不自动上传到服务器，除非用户主动执行"上传到云端"操作。

| 类型 | 位置 | 任何客户端可访问 | 云端保存 |
|------|------|------------------|----------|
| 云端项目 | 服务器磁盘 | ✅ | ✅ |
| 本地项目 | 用户电脑 | ✅（通过 SSH 隧道） | ❌（除非主动上传） |

**客户端能力矩阵**：

| 客户端 | 云端项目 | 本地项目 |
|--------|----------|----------|
| 浏览器 | ✅ | ❌（安全限制） |
| 桌面端（Tauri） | ✅ | ✅ |
| 移动端 | ✅ | ❌ |

#### .neuralswarm/ 目录

每个项目根目录下有一个 `.neuralswarm/` 目录，类似 `.git/`，存储项目配置、Spec 和记忆。所有文件均为纯文本格式（YAML、Markdown、JSONL），可通过 Git 管理。

```
project/
├── .neuralswarm/
│   ├── .gitignore        # 控制哪些提交、哪些不提交
│   ├── config.yml        # 项目配置（YAML）
│   ├── spec/             # 项目 Spec（Markdown）
│   │   ├── rules.md
│   │   └── architecture.md
│   └── memory/           # 项目记忆导出（JSONL）
│       ├── events.jsonl  # L1 事件流
│       └── knowledge.jsonl # L2 长期知识
├── src/
└── ...
```

**`.neuralswarm/.gitignore`**：
```gitignore
# 可重建的数据，不提交
cache/
embeddings/
logs/
*.bin
*.db

# 文本数据，提交到 Git
!config.yml
!spec/
!memory/*.jsonl
```

**config.yml 示例**：
```yaml
project:
  name: my-project
  description: "A web application"

agent:
  default_llm: claude-sonnet-4-6
  max_iterations: 20
  tools: [file_read, file_write, shell]

spec:
  auto_inject: true
  max_tokens: 4000
  storage: "neuralswarm"  # "neuralswarm" = .neuralswarm/spec/ | "git" = 项目根目录/spec/
```

**Spec 管理**：
- 存在 `.neuralswarm/spec/` 或项目根目录的 `spec/`（用户选择）
- Markdown 格式，可手动编辑
- Agent 执行任务前，根据任务描述检索相关 Spec 片段注入 System Prompt
- 支持和 Git 分支联动（不同分支不同 Spec）

**记忆与数据库双向同步**：
- 数据库是主存储，本地文件是可读可导出的镜像
- Agent 写入记忆后自动同步到本地文件
- 用户编辑 Spec 文件后自动导入到数据库
- 支持项目导入（文件 → DB）和导出（DB → 文件）

### 6.2 资源锁与冲突解决

- **资源标识**：采用URI范式，如 `project://neuralswarm-code/src/utils.py`。
- **锁模式**：
    - **独占锁（X锁）**：获取即排他。
    - **共享读锁（S锁）**：多读并发。
    - **意向锁（IX/IS）**：为目录等父级资源设计的锁，兼容更高层次操作。
- **分布式实现**：基于Redis的Redlock算法，保证在部分节点故障下锁的正确性。锁信息会持久化到PostgreSQL，用于崩溃恢复。

### 6.3 用户窗口调度器（Human-in-the-Loop）

当Agent A申请的X锁被Agent B持有时，系统不盲目阻塞，而是触发一个结构化的人机交互流程：

1.  **推送调度事件**：通过WebSocket向项目成员的悬浮窗推送一个JSON格式的 `scheduling_request`，包含冲突详情和上下文快照。
2.  **决策界面化**：悬浮窗展示Agent A、B的当前任务摘要、历史对话、锁持有时间及**推荐操作**（基于一个微调的决策模型）。
3.  **原子化决策执行**：用户选项包括：
    - **抢占**：保存B的检查点后终止B，锁交给A。
    - **优先级排队**：将A加入等待队列，并可动态提升A的优先级。
    - **任务合并**：若判定A的目标是B的子集，将A的任务指令和上下文注入B，让B一并执行。
    - **手动干预**：打开一个临时终端或编辑器，让用户快速解决问题后手动释放锁。
4.  **策略自动执行**：若超时未响应（默认5分钟），系统执行预设默认策略。

## 7. 能力分级与调度

调度器同时管理两层能力：**Agent 能力**（工具、项目上下文）和 **LLM 能力**（推理、代码生成）。任务分配是选 Agent + 选 LLM 的组合优化。

### 7.1 LLM 能力配置与在线校准

LLM 能力画像采用**声明式配置**，而非自动评估：

```yaml
llm_models:
  - id: claude-opus-4-6
    provider: claude
    gateway_route: claude-opus
    capabilities:
      codegen: 5
      reasoning: 5
      long_context: 5
    cost_tier: high
    rate_limit: { rpm: 30, tpm: 50000 }
    is_free_tier: false

  - id: claude-sonnet-4-6
    provider: claude
    gateway_route: claude-sonnet
    capabilities:
      codegen: 4
      reasoning: 4
      long_context: 4
    cost_tier: medium
    rate_limit: { rpm: 60, tpm: 100000 }
    is_free_tier: false

  - id: deepseek-v3
    provider: openai_compat
    gateway_route: deepseek
    capabilities:
      codegen: 4
      reasoning: 3
      long_context: 3
    cost_tier: low
    rate_limit: { rpm: 100, tpm: 200000 }
    is_free_tier: true
```

调度器定期拉取在线榜单（LMSYS Arena、HumanEval、MMLU 等）校准 LLM 能力分数，但不自动评估——用户声明的配置是第一优先级。

### 7.2 Agent 能力配置

Agent 能力由其工具集和项目上下文决定，与 LLM 无关：

```yaml
agents:
  - id: code-agent
    tools: [file_read, file_write, shell, git, test_runner]
    project_context: true
    memory_access: [L0, L1, L2]
    suitable_for: [codegen, refactoring, debugging]

  - id: review-agent
    tools: [file_read, git_diff, comment]
    project_context: true
    memory_access: [L0, L1, L2]
    suitable_for: [code_review, analysis]

  - id: chat-agent
    tools: [file_read]
    project_context: false
    memory_access: [L0]
    suitable_for: [question, explanation]
```

### 7.3 自动切换与 Token 均衡

当 LLM 不可用时，调度器自动降级：

1. **限流检测**：模型网关返回 429 或配额耗尽，调度器立即将该 LLM 标记为 `throttled`
2. **自动降级**：按 LLM 能力等级从高到低，选择下一个可用且未被限流的模型
3. **Token 消耗均衡**：调度器维护每个 LLM 的 token 消耗计数器，优先选择消耗较低的模型，避免把免费额度全砸在一个模型上
4. **恢复探测**：被限流的 LLM 每隔 N 分钟发送一次探测请求，恢复后重新加入调度池

```python
# 伪代码：LLM 选择逻辑
def select_llm(task, agent):
    candidates = filter_by_capability(llm_models, task.required_capability)
    candidates = filter(lambda m: m.status != 'throttled', candidates)
    candidates = sort_by_token_consumption(candidates)  # 消耗少的优先
    return candidates[0] if candidates else ask_user_for_downgrade(task)
```

### 7.4 任务匹配：Agent × LLM 组合

任务分配是选 Agent + 选 LLM 的组合优化：**最小化任务完成时间（含失败惩罚） × 最大化资源利用率。**

1. **任务解构与需求分析**：使用一个擅长意图理解的轻量级 LLM 将用户任务分解为子任务 DAG，并为每个子任务标注所需的 Agent 能力（工具集）和 LLM 能力（推理/代码生成等级）。
2. **匹配评分函数**：
    `Score(A, L, T) = agent_match(A, T) * llm_match(L, T) * g(load_A)`
    - `agent_match` 评估 Agent 的工具集是否覆盖任务需求
    - `llm_match` 评估 LLM 的能力是否满足任务复杂度
    - `g` 是负载惩罚函数，空闲 Agent 得分高
3. **动态扩容与降级**：若无合适 Agent+LLM 组合，调度器可触发**扩容**（创建新 Agent 实例并绑定不同 LLM）或**协商降级**（询问用户是否接受使用能力稍弱的组合）。

## 8. 统一Spec环境：项目的共享知识

Spec是项目的“法规”，经历版本化演进。为确保Agent行为一致：

- **Spec即代码**：存储在Git仓库，与代码分支/标签联动。当Agent执行基于`v2.0`分支的任务时，注入该分支的Spec。
- **向量化分块注入**：长篇Spec被分块并向量化。Agent任务执行前，根据其任务描述从Spec向量库中检索Top-K相关片段，注入其System Prompt。这解决了上下文窗口限制，并提供了最相关的规范指引。
- **变更影响分析**：当新Spec被推送，一个专门的“Spec-Change”事件被触发。调度器会评估所有进行中任务，将那些受影响的Agent标记为“待观察”，并向用户发送建议，如“重构Agent-A以遵循新API定义”。

## 9. 多层认知记忆架构

这模拟了人类的遗忘曲线和记忆巩固过程，是Agent长期进化的关键。

- **L0：瞬时记忆** —— 单个Agent的上下文窗口及函数调用历史。持久化于 PostgreSQL（Agent.context 字段），Redis 作为快速缓存。Agent 重启时可从 PostgreSQL 恢复上下文。
- **L1：情景记忆** —— 项目当前迭代的事件流（”成功修复了bug#442”，”重构了数据库连接池”）。持久化于 PostgreSQL（ProjectMemory, level=L1），带 TTL（默认 24 小时），过期后自动清理。同时写入 Redis Streams 用于实时订阅。类似 recap，记录项目当前状态。
- **L2：语义记忆** —— 项目的长期知识（”模块X的架构原理”、”函数Y的边界条件”）。存储于 PostgreSQL（ProjectMemory, level=L2），永久保存。L0 + L1 + 用户要求经**反思 Agent 自动整理**后提升至 L2。
- **L3：过程性记忆** —— 用户全局偏好与交互模式。存储于 PostgreSQL（GlobalMemory），永久保存。它像一个”元提示”，在任何项目任务开始时，被静默注入以个性化Agent行为。
- **RAG：检索增强** —— 项目文件的向量检索，存储于 Qdrant。独立于记忆系统，用于从项目文件中检索相关上下文。

**记忆存储架构**：

| 层级 | 存储位置 | 生命周期 | 作用域 |
|------|---------|---------|--------|
| L0 | PostgreSQL + Redis 缓存 | 随 Agent 生命周期 | Agent 级 |
| L1 | PostgreSQL + Redis Streams | 24h TTL | 项目级 |
| L2 | PostgreSQL | 永久 | 项目级 |
| L3 | PostgreSQL | 永久 | 全局级 |
| RAG | Qdrant | 永久 | 项目级 |

**记忆流转**：
1.  任务到来，从 L2 检索相关知识，从 RAG 检索相关文件上下文。
2.  执行中，核心事件和洞察暂存 L1。
3.  任务结束，反思 Agent 异步将 L0 + L1 + 用户要求整理为 L2 长期知识。

## 10. 全局聊天模式

全局聊天是一个特殊的”无项目”模式，参考 openclaw 的交互理念设计——个人助手、无固定项目绑定、可跨项目派发、不阻塞聊天任务。

### 10.1 设计原则

- **无项目绑定**：全局聊天不关联任何项目空间，是一个独立的对话环境
- **只派发不阻塞**：用户在全局聊天中可以向多个项目空间派发任务，派发后立即返回，任务在后台异步执行，不阻塞聊天
- **独立 Spec 和记忆**：全局聊天有自己的 L0-L3 记忆和 Spec，与项目空间完全隔离

### 10.2 与项目模式的关系

- 用户可在全局聊天中跨项目讨论，其对话历史成为 L1 短期记忆
- 当切换到项目模式时，全局对话的摘要（而非全部原始数据）被注入新 Agent 的 L0，作为”启动器”，赋予其连贯性
- 项目 Agent **无法**写入全局记忆，只能读取，彻底杜绝了项目间污染的可能
- 用户可手动将全局聊天中的某段对话”钉入”指定项目的 L1 记忆，实现跨上下文信息传递

## 11. 开发者体验：作为“驾驶舱”的悬浮窗

悬浮窗不是一个监控面板，而是一个**控制驾驶舱**。

- **多Agent任务飞行视图**：以3D力导向图或DAG流形式，立体展示当前所有Agent的依赖关系、执行进度、资源争用热点。
- **上下文穿透**：开发者可点击任一Agent，实时观察其“所思所想”（System Prompt + 当前对话栈），并可直接注入指令或进行微调。
- **Time Travel Debugging**：对于关键资源，系统记录了完整的变更日志。用户可拉取一个时间轴滑块，查看文件在Agent操作下的演进历史，并能回滚到任一快照。

## 12. 部署、运维与安全

### 安全纵深防御模型

- **传输层**：所有组件间通信强制mTLS和SSH隧道。
- **资源访问层**：MCP桥接网关执行基于角色的访问控制（RBAC）。`Agent-A`只能访问其项目路由下的资源和显式授权的客户端资源。
- **操作审计层**：所有Agent的工具调用和人工干预操作，均以结构化日志存入不可篡改的审计后端。
- **执行层**：客户端Agent的操作权限受本地操作系统白名单（如SELinux/AppArmor配置）限制。
