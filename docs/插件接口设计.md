# NeuralSwarm/Code 插件接口设计

> 本文档定义插件的**契约**——怎么写一个插件、怎么注册、数据长什么样。
> 架构理念见《白皮书》，官方插件的具体设计见《官方插件设计》，技术栈见《技术架构选型》。

---

## 1. 唯一原语

> **一个 handler 绑定到一个具名的点。流到达这个点时，handler 被调用，拿到在途上下文（ctx），返回 ctx 影响流向。**

插件 = 一个或多个 handler 的集合。没有"插件类型"，只有"绑在哪个点上"。

## 2. 点的固定词汇表

插件**只能往已有种类的点挂 handler，不能发明新点**。点的种类固定：

| 种类 | 具名点 | 流经的数据 |
|------|--------|-----------|
| 管道阶段 | `user-message` | 用户原始输入 |
| | `llm-prompt` | 构建中的 prompt（可注入 system / context） |
| | `llm-response` | LLM 返回（文本或 toolcall） |
| | `tool-execute` | 即将执行的 toolcall（`.before` / `.after` 子时机） |
| | `tool-result` | toolcall 执行结果 |
| 工具 | `tool:<name>` | 该工具的调用参数 → 返回结果 |
| UI | `ui:<slot>` | UI 事件 → 渲染/回传决策 |
| 生命周期 | `lifecycle:start` / `lifecycle:tick` / `lifecycle:stop` | 节点/会话生命周期事件 |

## 3. 注册契约：只声明两件可验证的事

插件注册时只声明内核**查得了**的事实——这是"声明必须可验证"铁律的落地，杜绝内核判断不出真假的盲标签：

```yaml
plugin: concurrency-control
version: 1.0.0
handlers:
  - point: tool-execute.before     # ① 绑在哪个点
    entry: check_conflict          # ② handler 入口
    filter: [file_write]           # 只对哪些 toolcall 生效（可选）
    before: [audit-log]            # 拓扑序依赖（可选）
    after: []
transport: native                  # native | grpc | mcp
```

**不允许的声明**（内核无法验证 → 禁止）：`affinity: owner`、`stateful: true`、`exec: on-entry` 之类。路由、状态位置、执行节点**全部从资源地址推导**（见第 6 节），不靠自报。

**唯一的额外声明：幂等性**——因为它内核推不出，且驱动真实行为（要不要自动重试）、后果可验证（声明幂等却产生重复副作用 = bug），且是协议固有概念（如 HTTP GET）：

```yaml
  - point: tool:file_write
    entry: write_file
    idempotent: false              # 重试不安全，故障转移时不自动重发
```

## 4. ctx：在途上下文对象

handler 收到 ctx、返回 ctx。ctx 是管道流动的唯一载体（内部 MessagePack 编码）：

```
Context {
  session_id      会话标识
  trace_id        跨节点追踪
  workspace       当前会话工作区视图（挂载 → 项目 → Owner）
  messages        对话历史
  prompt          构建中的 prompt（system / context 片段 / user）
  tool_calls      LLM 请求的 toolcall 列表
  tool_results    toolcall 执行结果
  capability      本次调用携带的能力令牌
  terminated      置 true 则中止后续 handler
}
```

handler 按需读写 ctx 的字段：记忆插件往 `prompt.context` 追加片段；并发 hook 读 `tool_calls` 决定放行/阻断；Executor 填 `tool_results`。

## 5. toolcall schema：符合 OpenAI / Anthropic 规范

`tool:<name>` 的 handler 暴露标准 function calling schema 给 LLM。参数里的路径是**工作区相对路径**，不含节点信息：

```json
{
  "name": "file_read",
  "description": "Read a file from the workspace.",
  "parameters": {
    "type": "object",
    "properties": {
      "path": { "type": "string", "description": "Workspace-relative path. Defaults to current working dir." }
    },
    "required": ["path"]
  }
}
```

路径有默认值（当前工作目录/当前挂载）。模型要访问当前主机工作区外的资源，必须显式传主机名等限定条件。MCP server 的 `tools/list` 暴露的工具被内核当作 `tool:*` handler 原生纳入。

## 6. 路由由内核推导，插件不参与

插件**不声明**自己在哪执行。内核每次调用时，从涉及的资源地址推导（理念见白皮书第 7 章）：

```
资源地址集合 = 载荷里的地址（如 path 经工作区映射 → project://X）
            ∪ handler 注册时绑定的资源（如 node://self/cache）
            ∪ 上下文相对地址（self、ui-session）
解析每个地址 → authority 节点：
  唯一 authority  → 路由到该节点
  无 authority    → 注册了该 handler 的节点间负载均衡
```

插件作者写的是**单机心智**：读 ctx、处理、写 ctx。"我在哪、上下游在哪"由引擎和路由层负责，松耦合走管道搬运，紧耦合因 bind 同一资源自动共址。

## 7. 三档 transport 接入同一注册流程

| transport | 接入方式 | handler 入口 |
|-----------|---------|-------------|
| **native** | 编译进内核 / 官方签名动态库（`libloading`） | Rust 函数实现 `Handler` trait |
| **gRPC** | 独立进程，注册时给 gRPC 地址 | 内核生成的 gRPC stub 调用 |
| **MCP** | 启动 MCP server 进程，`tools/list` 自省 | 内核 MCP adapter 转译 |

三者注册的 manifest 字段完全一致（第 3 节），区别仅 `transport` 值与 `entry` 的解释方式。gRPC 的 protobuf 接口定义将集中在独立的 `proto/` 目录（后续补充）。

## 8. UI 插件契约：插槽 = UI 的点

UI 是后端的镜像（理念见白皮书第 13 章）。骨架屏由内核提供，UI 插件往**固定插槽**填组件，**不能发明新插槽、不嵌套子插槽**。

### 8.1 固定插槽清单

| 插槽 | 位置 | 典型占用者 |
|------|------|-----------|
| `ui:sidebar` | 聊天区左侧 | 文件树/工作区视图、记忆面板、插件面板 |
| `ui:toolbar` | 聊天区顶部 | 模型选择器、项目/工作区切换器 |
| `ui:dialog` | 模态弹窗 | 冲突解决 diff 视图、令牌授权确认 |
| `ui:statusbar` | 窗口底部 | 当前项目、连接状态、节点信息 |
| `ui:message` | 消息流内 | 自定义消息块（如 diff、图表渲染） |

同一插槽可挂 0~N 个组件，按 `before`/`after` 排序。

### 8.2 UI 插件声明

```json
{
  "plugin": "concurrency-control",
  "slots": [
    { "slot": "ui:dialog", "component": "ConflictResolver", "events": ["tool-execute.conflict"] }
  ]
}
```

### 8.3 样式契约

组件**只能用内核暴露的 design token 与基础组件库**（沿用《前端设计规范》的 CSS 变量：`--color-*` / `--radius-*` / `--text-*` / `--shadow-*` 等四套主题），不许自带样式系统，保证全局视觉统一。

### 8.4 动态加载

入口节点连接 → 收集已启用插件的插槽声明 → 动态 `import()` 远程组件 → 强缓存到本地 → 渲染。Owner 节点的 UI 能力不影响入口渲染：UI 是入口的事，插件 UI 组件从其所在节点的 HTTP 服务拉取。

## 9. 插件可用性与降级（跨节点）

- 项目 `.neuralswarm/config.yml` 声明需要哪些插件；缺失时阶梯获取：域内 P2P → 插件市场 → 提示手动安装。
- 必需插件全域缺失 → 诚实报错（`command not found`），不静默跳过。
- 可选插件（`required: false`）缺失 → 跳过该 handler，管道继续，不报错。
- UI 组件缺失 → 插槽留空，不影响其他 UI。

这条降级规则对管道 handler 与 UI 组件分别独立生效（管道看 Owner/能力节点，UI 看入口节点）。
