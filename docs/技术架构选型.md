# NeuralSwarm/Code 技术架构选型 v2.0

> 本文档定义技术栈决策。架构理念见《白皮书》，插件契约见《插件接口设计》。
> **v2.0 推翻了 v1.0 的中心化 Client-Server 选型**（见末尾「已废弃」）。

---

## 0. 一句话定调

一个 **Rust 单体二进制**就是一个完整的神经元节点：内核（六机制）+ 官方插件（native）+ HTTP 服务（供 Web/桌面 UI）。下载即用，零外部服务依赖。社区插件通过 gRPC/MCP 接入，可以是任何语言。

---

## 1. 内核语言：Rust

| 维度 | 理由 |
|------|------|
| 跨平台 | 原生编译 Windows/macOS/Linux，单文件分发 |
| 零运行时 | 节点要能在树莓派到服务器各种机器上跑，不背运行时 |
| P2P 生态 | rust-libp2p 是去中心化网络的成熟实现 |
| 嵌入式存储 | rusqlite/sqlx + SQLite，无服务端进程 |
| 已有基础 | 现有 `core/` 已是 Rust（tokio + tungstenite） |

异步运行时用 **tokio**（已是现有依赖、Rust 事实标准）。

## 2. 插件传输：三档，统一注册

插件不是一种实现，而是"绑在点上的 handler"，**transport 是正交维度**（理念见白皮书第 11 章）。三档都实现同一个 `Handler` trait，管道引擎只见 trait object：

| Transport | Crate / 协议 | 用于 | 开销 |
|-----------|-------------|------|------|
| **native** | 进程内 `Arc<dyn Handler>` / `libloading` 动态库 | 官方插件（标准库） | 零序列化 |
| **gRPC** | `tonic` | 社区插件，跨进程/跨节点 | protobuf 序列化 + 网络 |
| **MCP** | `rmcp` / 自实现 stdio·SSE | 外部 MCP 工具生态 | MCP 协议 |

```rust
// 调用方只见 trait，不感知 transport
trait Handler: Send + Sync {
    async fn invoke(&self, ctx: Context) -> Result<Context>;
}
```

热重载：native 动态库用 `Arc` 原子替换（持旧 Arc 的请求跑完自然 drop）；gRPC/MCP 进程重启 + 上游重连。

## 3. P2P 网络：libp2p

| 需求 | libp2p 组件 |
|------|------------|
| 局域网节点发现 | mDNS |
| 广域网节点发现 | Kademlia DHT |
| 域级状态广播（Gossip） | GossipSub |
| NAT 穿越 | AutoNAT + Circuit Relay |
| 加密传输 | Noise |
| 流复用 | Yamux |

去中心化网络不自己造轮子——libp2p 就是这一层的事实标准。

## 4. 本地存储：SQLite

去中心化节点不需要中心化数据库。每个节点本地一个 SQLite 文件，存两类数据：

| 数据 | 说明 |
|------|------|
| **节点本地状态** | 不广播：文件哈希快照、索引缓存、MCP 连接元数据 |
| **域级共享状态副本** | Gossip 收敛：记忆、工作区配置、插件清单、对话历史（带版本向量 + LWW 字段） |

Crate：`sqlx`（编译期校验）或 `rusqlite`（轻量）。**不用 PostgreSQL、不用 Redis**——它们是中心化组件，与去中心化目标冲突。

## 5. 数据序列化

| 场景 | 选型 | 理由 |
|------|------|------|
| 管道内部 ctx 传递 | **MessagePack**（`rmp-serde`） | 小、快、无模式 |
| gRPC 接口 | **Protobuf**（`prost`/`tonic`） | 有模式，跨语言，版本可演进 |
| toolcall schema | **JSON Schema** | 符合 OpenAI/Anthropic function calling 规范 |
| 配置文件 | **YAML/TOML** | 人类可读，与 `.neuralswarm/` 一致 |

## 6. 内置 HTTP 服务：axum

节点内置 `axum`（tokio 生态，与内核共用运行时），对外提供：
- 静态服务 Web UI（Tauri/Vue 构建产物）
- REST + WebSocket 给浏览器/桌面壳

浏览器打开 `https://<节点>:port` 即完整 UI，无需安装。

## 7. 前端：沿用现有栈

| 层 | 技术 | 说明 |
|----|------|------|
| 桌面壳 | **Tauri v2** | 现有，~5MB，Rust 后端 + Web 前端 |
| 前端框架 | **Vue 3 + TypeScript** | 现有 `client/` |
| 设计系统 | 现有《前端设计规范》CSS 变量 | design token 契约，UI 插件遵循 |
| Markdown/diff | markdown-it + shiki | 富文本渲染（独立 IDE 的优势） |
| 图标 | lucide-vue-next | 现有，零 emoji |

前端**不需要重写**，需要新增：骨架屏 shell（内核提供布局 + 插槽位置 + 动态加载）+ UI 插槽系统（插件往固定插槽填组件）。

## 8. 可观测性

| 用途 | 选型 |
|------|------|
| 结构化日志 | `tracing`（现有依赖） |
| 跨节点追踪 | 一次请求跨多节点，ctx 携带 trace-id，各节点 `tracing` span 关联 |

## 9. 节点二进制全景

```
┌─────────────────────────────────────────────────┐
│  neuralswarm 节点（单 Rust 二进制）              │
├─────────────────────────────────────────────────┤
│  内核（六机制）                                  │
│  点注册表 · 管道引擎 · 传输适配 · 寻址路由       │
│  · 域成员&Gossip(libp2p) · 令牌闸门              │
├─────────────────────────────────────────────────┤
│  官方插件（native，每节点自带）                  │
│  朴素LLM · 基础工具 · 记忆 · Spec · 并发控制 …   │
├─────────────────────────────────────────────────┤
│  SQLite（本地状态 + 域级状态副本）               │
├─────────────────────────────────────────────────┤
│  axum HTTP/WS（服务 Web UI + 浏览器接入）        │
└─────────────────────────────────────────────────┘
        ↕ gRPC(tonic)        ↕ MCP(stdio/SSE)
   社区插件（任意语言）   外部 MCP 工具生态
        ↕ libp2p
   信任域内其他节点
```

## 10. 现有代码迁移方向

> 仅指方向，不含步骤——具体迁移是执行层的事。

| 现有代码 | 处置方向 |
|---------|---------|
| `core/`（Rust：tungstenite 客户端、executor、mcp） | 演进为内核基础，executor/file 操作可复用为 native 工具插件的底层 |
| `server/`（Python：FastAPI/LLM/工具/并发/记忆） | LLM 适配器逻辑、工具实现、并发哈希逻辑作为参考迁入对应 native 插件；FastAPI/调度器/PostgreSQL/Redis 整体废弃 |
| `server/.../memory/`（L0-L3） | 废弃四层架构，重做为"便签本"记忆插件（见《官方插件设计》） |
| `server/.../scheduler/`（CentralScheduler 等） | 废弃，默认串行；高级调度后期作社区插件 |
| `client/`（Vue） | 保留，新增骨架屏 + UI 插槽系统 |
| `desktop/`（Tauri v2） | 保留 |

## 11. 已废弃（v1.0 → v2.0）

| v1.0 选型 | 废弃原因 |
|----------|---------|
| Python/FastAPI 服务端为"脑" | 中心化 Client-Server，与去中心化对等网络冲突 |
| PostgreSQL | 中心化数据库，节点不需要 |
| Redis（Streams/Pub-Sub） | 中心化消息，由 libp2p Gossip 替代 |
| RBAC + mTLS 中间件 | 账户中心安全模型，由能力令牌替代 |
| MCP over WebSocket 为唯一通道 | MCP 降为 transport 三档之一 |
| 多 Agent 中央调度器 | 过度设计，默认串行 |
